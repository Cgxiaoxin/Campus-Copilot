# Campus Copilot — AI 校招操作系统（PRD）

## Problem Statement

应届生在校招季面临多条招聘线并行投递的局面：每家企业、岗位的进度不同，网申表单重复填写耗时巨大，截止日期与笔面试时间容易遗漏，同一岗位需要多版本简历匹配不同 JD，且缺少从投递到 Offer 的整体规划工具。现有解决方案要么只解决单点问题（如简历优化），要么面向社招人群，缺乏对校招场景的垂直覆盖。

## 目标用户

- **大三学生**：寻找暑期实习，首次接触校招流程
- **大四学生**：参与秋招/春招，多条招聘线并行
- **硕博应届毕业生**：投递节奏更紧凑、目标更集中
- **核心场景**：每人同时投递 10-50 家企业，管理 3-8 个岗位方向

## 解决方案

Campus Copilot 是一款面向应届生的 AI 校招操作系统，以"一份档案（One Profile），多次高质量投递（Multiple Applications）"为核心理念，覆盖从岗位发现到 Offer 管理的全生命周期。

## User Stories

1. 作为一名应届生，我希望在一个地方统一管理我的教育、项目、实习经历，这样我不用每次投递都重复填写。
2. 作为一名应届生，我希望 AI 能根据我的档案和岗位 JD 自动生成网申表单内容，这样我能节省大量填表时间。
3. 作为一名应届生，我希望在一个看板上看到所有投递企业的进度状态，这样我能一目了然地掌握全局。
4. 作为一名应届生，我希望系统能根据截止日期自动生成每日任务并排序，这样我不会错过任何重要节点。
5. 作为一名应届生，我希望 AI 能根据岗位描述自动调整我的简历关键词和项目描述，这样我能提高简历匹配度。
6. 作为一名应届生，我希望系统能根据我的背景和岗位要求给出匹配评分，这样我能优先投递匹配度高的岗位。
7. 作为一名应届生，我希望 AI 能生成针对目标公司和岗位的模拟面试题，这样我能有针对性地准备面试。
8. 作为一名应届生，我希望在模拟面试后能获得详细的复盘报告和改进建议，这样我能持续提升面试表现。
9. 作为一名应届生，我希望系统能自动提醒我各企业的笔试、面试和截止日期，这样我不会遗漏任何关键时间点。
10. 作为一名应届生，我希望 Dashboard 能展示我本周的任务、投递进度和面试安排，这样我能合理安排每天的时间。
11. 作为一名应届生，我希望我的档案支持多种格式导入（如已有简历解析），这样我不需要从头手动录入。
12. 作为一名应届生，我希望在获得 Offer 后能记录和比较不同 Offer 的薪资待遇等信息，这样我能做出更明智的选择。

## 核心模块

### 1. Career Profile — 统一求职档案

统一维护教育背景、项目经历、实习经历、技能标签、获奖信息等。所有 AI 能力以此为数据基础。

### 2. AI 网申助手

解析网申字段，依据档案与 JD 自动生成填写内容，支持一键复制以及未来浏览器插件自动填充。

### 3. 校招作战看板

以看板形式统一管理企业、岗位、投递状态（草稿→已投递→筛选中→面试→Offer→未通过），展示下一步任务和截止时间。

### 4. Time Agent — 智能时间管理

自动生成每日计划，按截止时间和优先级排序，支持动态调整和推送提醒。

### 5. AI 简历定制

根据岗位 JD 自动优化简历关键词、项目描述表述和技能排序，提高简历与岗位的匹配度。

### 6. 投递策略引擎

基于用户背景与岗位要求进行匹配评分，给出投递建议和优先级排序。

### 7. AI 面试准备

针对目标公司和岗位生成模拟面试题，支持回答评估，输出复盘报告和改进计划。

## 实施决策

- **前端**：Next.js (App Router) + React + TypeScript + Tailwind CSS。使用 App Router 文件系统路由，每个模块对应独立路由页面。
- **后端**：FastAPI 提供 RESTful API。采用微服务风格但不强制拆分，初期以模块化单体架构启动。
- **数据库**：PostgreSQL 存储核心业务数据。使用 SQLAlchemy 作为 ORM。
- **缓存与队列**：Redis 缓存热点数据，Celery 处理异步任务（如简历生成、面试题生成等耗时操作）。
- **AI 架构**：统一 LLM 调用层，支持 Qwen / DeepSeek / GPT 切换。使用 Embedding + 向量检索实现 RAG 增强。Multi-Agent 编排处理复杂流程（如简历定制流程需同时分析 JD、检索档案、调整内容）。
- **认证**：JWT 令牌认证，支持邮箱登录和第三方 OAuth。
- **API 设计**：RESTful JSON API，版本化（/api/v1/...）。每个 Agent/Service 对应独立路由模块。

## 测试策略

- **测试分层**：单元测试（Service 层）→ 集成测试（API 端点）→ E2E 测试（核心用户流程）
- **前端测试**：使用 Vitest + React Testing Library 测试组件行为，而非实现细节
- **后端测试**：使用 pytest + httpx (AsyncClient) 测试 API 端点
- **AI 测试**：使用固定 prompt 和 mock LLM 响应进行确定性测试
- **E2E 测试**：使用 Playwright 覆盖核心用户路径（创建档案→生成简历→添加投递→管理任务）
- **测试优先**：新功能应优先编写测试定义预期行为，再实现功能代码

## 范围外（Out of Scope）

- 浏览器自动填表插件（MVP 后迭代）
- 移动端 App（先专注 Web 端）
- 企业端 / HR 后台（纯面向学生）
- 社区 / 面经分享功能
- 岗位爬虫与自动发现（MVP 依赖用户手动添加）

|------|--------|------|
| Career Profile | P0 | 核心数据基础 |
| 校招 Dashboard | P0 | 全局概览 |
| AI 简历生成 | P0 | 核心差异化功能 |
| AI 网申助手 | P1 | 字段填充与复制 |
| Time Agent | P1 | 任务管理 |

## 进一步说明

- 差异化定位：聚焦应届生校招全生命周期管理，而非单纯简历优化工具
- 扩展性：数据模型和 API 设计需预留未来扩展空间（如插件系统、多端同步）
- 数据安全：用户档案数据属于敏感信息，需做好数据加密和访问控制
